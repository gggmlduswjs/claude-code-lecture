

팀 하네스 공유

클로드 md를 깃배포에 커밋 

my-project/
├── CLAUDE.md          ← 팀 공유 헌법 (Git 커밋)
├── .claude/
│   ├── rules/         ← 경로별 세부 규칙 (Git 커밋)
│   └── settings.json  ← Hooks 정의 (Git 커밋)
├── src/
└── ...

claude.md에 들어가는 것
- 기술 스택&아키텍처 개요
- 빌드, 테스트, 린트 명령어
- 코딩 컨벤션(네이밍,임포트,에러처리)
- 절대하면 안되는 것들 

.claude/rules/ - 세분화된 규칙

# .claude/rules/api-routes.md
---
paths: ["src/api/**/*.ts"]
---
API 라우트를 작성할 때:
- 모든 입력에 zod 스키마로 유효성 검증을 추가할 것
- 응답 형식은 항상 { data, error, meta }를 따를 것
- 에러 발생 시 적절한 HTTP 상태 코드를 반환할 것
# .claude/rules/tests.md
---
paths: ["**/*.test.ts", "**/*.spec.ts"]
---
테스트 파일을 작성할 때:
- vitest를 사용할 것 (jest 사용 금지)
- describe/it 패턴을 사용하고 한글로 테스트명을 작성할 것
- 각 테스트는 독립적으로 실행 가능해야 함

.claude/hooks - 팀 자동화
- 자동 포맷팅 : 클로드가 파일을 수정하면 자동으로 prettier 실행. 팀원이 기억할 필요 없이 팀원이 보장(posttooluse -> prettier--write)

Claude Code의 "훅(hooks)" 기능 얘기예요. 한 줄로: **"클로드가 어떤 동작을 하면, 그 직후에 내가 정해둔 명령을 자동으로 실행"**시키는 장치.

뜯어보면
PostToolUse = "클로드가 도구(툴)를 쓴 직후"라는 타이밍. 여기선 클로드가 파일을 편집(Edit/Write)한 직후.
prettier --write = 코드 포맷 정리 도구 (들여쓰기·따옴표·줄바꿈 통일).
합치면: 클로드가 파일 고칠 때마다 → 자동으로 prettier가 돌아서 코드 모양을 깔끔하게 맞춰줌.
왜 "팀 자동화"냐
이 훅 설정은 repo 안(.claude/settings.json)에 들어가서 git으로 공유돼요. 그래서:

팀원 A, B, C 누구도 "아 포맷 돌리는 거 까먹었다" 할 일이 없음.
사람이 기억/의지로 지키는 게 아니라 시스템이 강제로 보장함.
원래는 커밋 전에 사람이 prettier 수동으로 돌려야 했는데, 그걸 클로드 동작에 자동으로 묶어버린 거예요.

설정 생김새 (예시)

{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{ "type": "command", "command": "prettier --write \"$file\"" }]
    }]
  }
}
→ "Edit나 Write 툴을 쓰면, 방금 건드린 파일에 prettier --write 돌려라"
