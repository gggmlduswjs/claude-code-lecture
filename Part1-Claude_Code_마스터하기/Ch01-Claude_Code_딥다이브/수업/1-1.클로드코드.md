

클로드 코드에 작업을 주면 3단계를 반복한다

컨텍스트 수집 -> 작업 수행 -> 결과 검증

1. 컨텍스트 수집
- glob(파일찾기),grep(내용검색),read(읽기),bash(명령실행)


2. 작업 수행
- edit(파일편집), write(쓰다), bash(명령실행), mcp(클로드를 외부 서비스랑 연결하는 규칙)

3. 결과 검증
- bash(명령실행-테스트돌리기), read(읽기), test(검사)


┌──────────────────┐
│ ① 컨텍스트 수집  │  파일 탐색·읽기, 명령 출력 확인

         ↓
┌──────────────────┐

         ↓

         ↓
     완료 또는 ① 로 복귀

diff(디프) = "difference", 즉 **바뀐 부분(차이)**이에요. 코드가 이전 상태에서 지금 상태로 뭐가 추가되고 뭐가 지워졌는지만 딱 보여주는 것.

diff 생김새

  def create_order(items):
-     total = 0
+     total = sum(i.price for i in items)
      return total
- = 지워진 줄 (빨강)
+ = 추가된 줄 (초록)
나머지(색 없는 줄) = 안 바뀐 부분 (맥락용)
즉 파일 전체가 아니라 "달라진 데만" 보여주는 게 diff. (git diff 명령으로 봄)

"diff 단위 작업"이란
작업을 한 덩어리로 크게 하지 말고, 작고 목적이 하나인 변경(diff)들로 쪼개서 진행하라는 뜻이에요.

❌ 나쁜 예: "화면 개편 + 버그 수정 + 리팩토링"을 한 번에 왕창 → diff가 500줄
✅ 좋은 예:
diff ①: 버그 한 개 수정 (10줄)
diff ②: 리팩토링만 (50줄)
diff ③: 새 기능 (80줄)
각 diff가 **"딱 한 가지 일"**만 하게.

왜 그렇게 하냐
리뷰가 쉬움 — 작으니까 뭐 바뀌었는지 한눈에 봄
사고 추적 쉬움 — 문제 터지면 "어느 diff에서 깨졌지" 바로 찾음
되돌리기 쉬움 — 그 diff만 revert하면 됨
