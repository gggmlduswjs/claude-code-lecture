

에이전트가 했다고 하면 누가 검증합니까?

Closed-loop validation — 캐주얼한 prompting과 프로페셔널 엔지니어링을 가르는 1차 differentiator (Boris Cherny).

1. self-correction : 에이전트 스스로가 수정한다
2. loop : 한 번이 아니라 반복 구조 : 실패 -> 분석 -> 수정 -> 재시도(통과할 때까지), 사람의 개입이 없음

에이전트한테 인프라가 잇어야함

나쁜출력(Crappy Output)
잘된 출력(Validated Work)

확률 엔진은 자기가 만든 결과를 자기가 평가하면, 그 평가도 같은 확률 분포의 영향을 받는다. 자기 평가는 신뢰할 수 없다 — 신호는 모델 밖에서 와야 한다.

공통 패턴 — 외부 시스템에 작업을 실행시키고, 결과 신호를 agent context에 다시 넣는다. 스택은 달라도 구조는 같다.


1차
[Pass/Fail ]
- 결제 API가 200 OK인데 minor unit 변환 버그로 **100배 청구**
- 결제 성공인데 영수증 이메일 누락
- DB에는 기록됐는데 외부 PG와 reconciliation 깨짐
- 응답은 정상인데 비정상 retry가 로그를 채움

2차
[ Observability-Driven Validation]
내부 system state 가 의도한 결과와 일치하는지 검증
(데이터무결성)

1. agent가 작업 완료
2. unit/integration test 통과 (pass/fail 검증) — 1차 layer
3. LogQL 쿼리로 내부 state 검증 — 2차 layer
4. 둘 다 통과해야 "validated work"로 인정
5. 어느 하나라도 실패 → self-correction loop 진입
