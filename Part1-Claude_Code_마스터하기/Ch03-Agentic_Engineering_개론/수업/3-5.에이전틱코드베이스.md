Brownfield의 4단계 독성 작용

1. Tech Debt 축적     
— src/billing/ vs src/payments/, lib/v1 vs lib/v2 vs lib/old
2. Context 오염        
— agent가 모든 디렉토리를 동등하게 context에 넣음
3. Hallucination 또는  
— agent가 deprecated 패턴 채택 또는
   Deprecated 패턴 채택   존재하지 않는 제3의 함수 호출 fabricate
4. Variance 폭발       
— 50% × 50% × 50% × ... → "AI는 일관성이 없다"


같은 일을 하는 두 가지 방법이 코드베이스에 공존한다면
- 에이전트가 잘못된 쪽을 고를 확률은 50%다

해결책: "relentless cleanup of competing patterns." relentless가 핵심 — 새 경쟁 패턴이 생기는 즉시 정리, 지속적 detoxification.

우리가 환경세팅을 제대로 못한 잘못

[consistency over dry]


dry : 같은 코드를 여러 군데에 놓지 마라(옛날방식)
solid:

ai한테 추상화는 컨텍스트 비용을 늘림



// DRY 버전 (추상화) — agent는 정의 찾아가야 함

중요 nuance — "DRY/SOLID 폐기"가 아니라 "맹목적 적용 중단"

- **Foundational** — 여전히 기초, 폐기 X
- **Evolving** — *언제 적용할지의 판단 기준*이 진화
- **Extreme adherence** — *극단적 고수*가 문제. 적절한 적용은 여전히 좋다

[에이전트옵티미즈코드베이스]

1	Elimination of Pattern Contamination	dead code · deprecated logic · 미완성 마이그레이션 제거
2	Structural Consistency	디렉토리 구조·네이밍 컨벤션의 rigid·predictable 유지
3	Agent-Specific Logging	agent 소비용 machine-readable 구조화 로그
4	Golden Principles / Rules	아키텍처 표준과 opinionated rules를 RAG-친화 마크다운으로 인코딩

# Golden Rules — 절대 위반 금지

1. 결제 관련 코드는 src/payment/에만. billing/, pay-utils/는 DEPRECATED.
2. 외부 API 호출은 항상 lib/clients/ wrapper 경유. 직접 fetch 금지.
3. 통화는 minor unit으로만 저장. major unit 변환은 출력 직전에만.
4. 모든 결제 액션은 logger.payment 사용. console.log 금지.
5. 새 dependency 추가는 architect 승인 필요.
